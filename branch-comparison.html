03 · Why the change was required

The repo had drifted from its own decisions

1

Structure ≠ stack

The specs already said “one Python FastAPI backend.” The disk still showed apps/api, apps/orchestrator, apps/ai-services — five split shells, mostly empty.

2

Nothing ran

There was no executable backend — only placeholders. You couldn't uv run, couldn't test, couldn't gate a PR on anything real.

3

Docs + debt lied

Architecture docs still named EKS, Prisma, NestJS and tenant_id; the repo tracked node_modules and junk test files. New contributors would trust the wrong map.

The stack was already decided. The repository simply hadn't caught up to it.

Why a modular monolith? The agent/ML ecosystem (LangGraph, LlamaIndex, DSPy, TruLens) is Python-native — splitting it across services adds subprocess/HTTP latency on the hot path. One deployable now; split back into ECS services in Phase 4 only if scale demands it.

11 · The decisions, with reasons

Reconciliation rationale

DecisionWhy it was made
One FastAPI backendapps/* → backend/appThe whole agent/ML ecosystem is Python-native; cross-language bridging adds latency & ops complexity on the hot agent path.
Modular monolith, for nowsplit deferred → Phase 4One thing to build, test, and deploy. Re-split into separate ECS services only when scale actually requires it.
Supabase platformPG + pgvector + auth + storage + realtimeCollapses RDS + Qdrant/Pinecone + Auth0 + S3 into one hosted platform — far less operational overhead.
ECS Fargate, not EKSCodeDeploy blue/greenNo Kubernetes control-plane to operate for v1; serverless containers + managed blue/green deploys.
